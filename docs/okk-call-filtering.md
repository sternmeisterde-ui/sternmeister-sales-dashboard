### Что оценивает OKK vs что показывает дашборд

OKK-сервис ставит `prompt_type` **по стадии** lead'a в Kommo (status-based routing — см. `/Users/user/okk/src/config/constants.ts`). Дашборд группирует и фильтрует строки **по `prompt_type`** — каждая UI-кнопка таба = один промпт, чтобы криатерия-таблица всегда соответствовала JSON-схеме своего промпта (`src/criteria/d2_*.json`).

4 промпта = 4 кнопки в UI (Линия 2 концептуально объединяет Бератер 1 и Бератер 2, но они всегда показаны как отдельные кнопки):

| Кнопка | prompt_type | JSON-схема |
|---|---|---|
| Квалификатор | `d2_qualifier` | `d2_qualifier.json` |
| Бератер 1 | `d2_berater` | `d2_berater.json` |
| Бератер 2 | `d2_berater2` | `d2_berater2.json` |
| Доведение | `d2_dovedenie` | `d2_dovedenie.json` |
| Все | (нет фильтра) | funnel-row по prompt_type |

### Known issue (OKK side, не дашборд)

¹ По исходной задумке стадии «Консультация перед термином ДЦ» и «проведена» должны идти в `d2_dovedenie` для менеджеров доведенческой роли (Кристина+Лихварь, `managers.line='3'`). По факту OKK ставит им `d2_berater` (status-based routing без manager-line override). В результате в табе **Доведение** дашборда видны только звонки на стадии «Доведение» — звонки тех же менеджеров со стадий «Конс ДЦ» уходят в **Бератер 1**.

Это не фиксится на стороне дашборда (фильтр по managers.line даёт mismatch имён блоков в evaluation_json и в d2_dovedenie.json — таблица criteria рисует прочерки). Решение — на стороне OKK: либо переоценить «Конс ДЦ» звонки доведенческих менеджеров по d2_dovedenie, либо добавить manager-line override в `getPromptTypeByPipelineId`. См. `/tmp/okk-cleanup/SPEC-UPDATE-PROPOSAL.md` секция Variant A.

## Какие поля берутся из Kommo

| Поле | Откуда | Для чего |
|---|---|---|
| Контакт (имя, ID) | `/contacts?query=телефон` | Привязка к контакту |
| Сделка (ID, название) | Через контакт → leads | Привязка к сделке |
| Воронка | `lead.pipeline_id` | Определение отдела (R2/D2) |
| Стадия | `lead.status_id` | Определение типа оценки |
| Ответственный | `lead.responsible_user_id` | Привязка к менеджеру |
| Ссылка на CRM | `leads/detail/{leadId}` | Ссылка в дашборде и Telegram |
| Custom fields | Только для D2 | Доп. информация по клиенту |

### Поиск контакта по телефону

1. Ищем контакт по номеру → берём все его сделки
2. Если сделок нет → ищем сделки напрямую п�� номеру
3. Из всех сделок выбираем лучшую:
   - Активная в известной воронке (приоритет)
   - Любая в известной воронке
   - Любая активная
   - Последняя по дате
